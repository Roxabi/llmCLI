## Problem

Step 3 of the central-portal plan (#40) shipped `llmcli proxy` as a foreground binary that owns both config generation and process lifecycle. Step 4 now packages that binary as a Podman Quadlet so user-systemd supervises it directly on both M‚ÇÅ (prod) and M‚ÇÇ (dev) ‚Äî finally allowing the `:4000` lyra-supervisor instance to be retired (Step 6).

Trigger: Step 3 is merged; consumers (lyra, hermes, claude-code) need a stable, restart-survivable `:18091` endpoint that survives reboots without manual `make llm`.

Observable today: `llmcli proxy` must be started manually on both hosts (`make llm` or shell). Crash on M‚ÇÅ requires manual intervention. M‚ÇÇ has no clean lifecycle (`make llm` blocks the shell).

## Who

- **Primary:** Mickael (ops) ‚Äî needs `systemctl --user start/restart/status llmcli` to manage the portal on both hosts with the same grammar
- **Secondary:**
  - lyra, hermes, claude-code consumers ‚Äî gain a stable, restart-survivable `:18091`
  - M‚ÇÅ (roxabituwer) ‚Äî finally drops lyra-supervisor as a hard dep for proxy hosting
  - Future Quadlet adopters (voiceCLI, imageCLI) ‚Äî `llmcli.container` becomes the reference pattern

## Constraints

- Quadlet `.container` unit only ‚Äî no Compose, no Kubernetes (lyra/voiceCLI pattern)
- M‚ÇÅ (roxabituwer, prod): autostart via user-linger; restart on crash
- M‚ÇÇ (roxabitower, dev): manual `systemctl --user start`, no auto-restart on crash
- Mount: `~/.roxabi/llmcli/` (catalog, RW) + `~/.local/state/llmcli/` (state, RW) + `~/.cache/huggingface/` (model cache, RW)
- Env: `LLMCLI_API_KEY`, `LLMCLI_PROXY_PORT=18091`, `LLMCLI_PROXY_HOST=0.0.0.0`
- Image: reuse llmcli's published image OR build local from `pyproject.toml` if none exists (decide in spec)
- `enable` is a no-op for Quadlet-generated user units ‚Äî persistence is auto-wired by the generator (per project memory: `feedback_quadlet_enable_is_noop.md`)
- Python 3.12, `uv` + `hatchling` (existing stack ‚Äî no new build runtime)

## Premise Validity

**Success in 6 months:** `systemctl --user status llmcli` is green on M‚ÇÅ and M‚ÇÇ; the unit survives M‚ÇÅ reboots (validated via at least one observed reboot); consumers (lyra, hermes, claude-code) transit `:18091`; the `:4000` lyra-supervisor proxy is retired and the `litellm.conf` removed from `~/projects/lyra/deploy/supervisor/conf.d/`.

**Failure in 6 months:** ‚â•1 prod incident per month on M‚ÇÅ where the Quadlet fails to start or restart and requires manual `podman` intervention, OR M‚ÇÇ never adopts the Quadlet (still using `make llm` 6 months in). Either condition is a falsifiable rollback signal ‚Äî observable via `journalctl --user -u llmcli` (start/restart failures) and via the absence of `llmcli.service` in `systemctl --user list-units` on M‚ÇÇ.

**Simplest alternative:** Raw systemd `.service` unit (no Quadlet) wrapping `podman run ‚Ä¶` manually ‚Äî declares the same restart policy, mounts, env, and port via standard `[Service]` directives.
**Why not simplest:** Re-invents what Quadlet's generator already does ‚Äî image refresh hooks (`AutoUpdate=registry`), volume + network plumbing, dependency wiring (`Requires=`/`After=`). Bare-`.service` units drift between M‚ÇÅ and M‚ÇÇ because each host's `podman run` flags get edited independently. Quadlet's declarative `.container` format keeps both hosts on the same source-of-truth; the rest of Roxabi (lyra, voiceCLI) is already on this pattern, so reusing it pays a one-time learning cost across the fleet.
