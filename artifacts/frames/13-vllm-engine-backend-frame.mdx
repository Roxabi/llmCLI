## Problem

llmCLI serves local models exclusively through llama-server binaries (vanilla and TurboQuant fork), both of which require GGUF format. A growing class of quantized models — particularly NVFP4 and GPTQ variants distributed as safetensors — is only supported by vLLM. Without a vLLM engine, these models cannot be served through llmCLI at all.

The immediate blocker is `kaitchup/Qwen3.6-27B-autoround-nvfp4-linearattn-BF16`: an NVFP4-quantized 27B model that fits on the RTX 5070 Ti (~15 GiB VRAM) and offers 262K context with reasoning support, but cannot be loaded by llama-server.

vLLM is already acknowledged in the CLAUDE.md engine table as a planned optional extra (P7). This issue makes it real.

## Who

- **Primary:** llmCLI operator (Mickael) — wants to run vLLM-only models on the dev machine via the same `llmcli serve` / `llmcli swap` interface used for llamacpp models
- **Secondary:** Lyra agents and Claude Code (via LiteLLM proxy) — consume the OpenAI-compatible endpoint; engine swap is transparent to them

## Constraints

- vLLM v0.19+ required (NVFP4 quantization support)
- `transformers` must be installed from GitHub HEAD (HF hub lags behind vLLM's requirements)
- Dev-only: RTX 5070 Ti (16 GiB) — prod RTX 3080 (10 GiB) is too small; no prod catalog entry
- Must implement the existing `Engine` protocol (`start`, `stop`, `health`, `base_url`) so the daemon, swap, and status commands work without modification
- `flags` passthrough must work (e.g. `--max-model-len`, `--reasoning-parser qwen3`)
- vLLM is an optional dependency group — `uv sync` without `--group vllm` must still work

## Out of Scope

- Multi-GPU / tensor-parallel serving
- vLLM on prod (roxabituwer / RTX 3080)
- Automatic vLLM install — user installs manually (`uv pip install vllm`)
- Migrating existing llamacpp models to vLLM
- vLLM-specific features (speculative decoding, LoRA, etc.)

## Complexity

**Tier: F-lite** — single domain (engine layer), clear interface contract already established by `llamacpp.py` and `llamacpp_tq3.py`, no architectural changes required.

Signals:
- Files touched: ~5–7 (vllm.py, engine.py factory, pyproject.toml, llmcli.example.toml, CLAUDE.md, 1–2 test files)
- Technical risk: low — subprocess management pattern is established; vLLM health endpoint is standard
- Architectural impact: additive only — new engine registered in factory, no existing code changes
- Unknowns: vLLM subprocess lifecycle (startup time, health check endpoint path) — minor, verifiable
- Domain breadth: 1 (engine layer)
