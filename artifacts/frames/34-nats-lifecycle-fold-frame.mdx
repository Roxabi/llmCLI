## Problem

llmCLI carries **two control planes per GPU host**:

| Plane | Transport | Owner | Reach |
|---|---|---|---|
| Data (inference) | NATS `lyra.llm.generate.request` | `LlmNatsAdapter` (quadlet, host network) | tailnet-wide via nkey |
| Control (lifecycle) | AF_UNIX socket at `~/.local/state/llmcli/llmcli.sock` | `src/llmcli/daemon.py` (host process) | local-only, requires SSH |

The split has three concrete recurring costs (per ADR-004 ¬ßContext):

- **Two deploy patterns per host.** Quadlet for the worker, manual / supervisor for the daemon. M‚ÇÇ has no clean answer today.
- **Location asymmetry.** Inference works tailnet-wide; lifecycle requires SSH to the GPU host. `llmcli swap` from M‚ÇÇ ‚Üí M‚ÇÅ doesn't exist; operators `ssh roxabituwer` then run `llmcli swap`.
- **Two auth/observability surfaces.** NATS nkey + JetStream metrics for one plane; filesystem perms + ad-hoc logs for the other.

Surrounding plumbing has matured in the 12 days since #34 was filed and *strengthens* the case:

- ADR-004 itself was committed post-filing (`44d21b2`), making the architectural premise explicit.
- The worker quadlet shipped (#44, `7bc0e8d`) ‚Äî the "existing quadlet worker" the issue assumes is now real on disk at `deploy/quadlet/llmcli-nats-worker.container`.
- `LlmNatsAdapter._ensure_model` already has a graceful "daemon socket absent ‚Üí assume external load" fallback (`src/llmcli/nats/llm_adapter.py:100-107`), providing a soft migration cushion.
- `cli/lifecycle.py status` already parses both `OK model=‚Ä¶` plain text **and** JSON dict payloads ‚Äî already forward-compatible with richer responses.

Without folding, a 3rd GPU host pays the duplication tax again, the SSH asymmetry persists, and the worker keeps its half-life dependency on a daemon process the rest of the lyra ecosystem doesn't need.

## Who

- **Primary:** Mickael (operator) on M‚ÇÇ wanting to manage M‚ÇÅ's loaded model without SSH. Today: `ssh roxabituwer; llmcli swap qwen3-4b`. Goal: `llmcli swap --host roxabituwer qwen3-4b` from any tailnet member.
- **Secondary:**
  - **lyra agents** ‚Äî consume llmCLI via `LlmNatsAdapter`, unaffected functionally; benefit from unified observability (lifecycle audit trail in JetStream).
  - **claude-code (ccl/ccp aliases)** ‚Äî consume llmCLI via LiteLLM `:4000` (and post-#51, via the `:18091` quadlet proxy); fully unaffected ‚Äî both proxies stay in place. `chat` command and inference data plane untouched.
  - **Future GPU host (3rd machine)** ‚Äî pays only the quadlet deploy cost, not the daemon supervisor wiring.

## Out of Scope

- **Removing LiteLLM `:4000`** ‚Äî claude-code still consumes it via `ccl`/`ccp`. Retirement is #51, gated separately.
- **Rewiring `llmcli chat`** ‚Äî already bypasses lifecycle (plain OpenAI HTTP to LiteLLM). Unaffected.
- **Replacing the data-plane streaming pattern** ‚Äî `GenerationMixin._stream_response` chunk-publish-on-reply-inbox stays as-is; data and control namespaces are disjoint.
- **Exposing `llama-server` directly from the worker** (skipping LiteLLM for inference) ‚Äî separate decision; not load-bearing here.
- **Cross-repo NATS subject schemas for other domains** ‚Äî only `roxabi_contracts.llm.*` subjects added.
- **Operator nkey provisioning UX** ‚Äî assumed already operable for the human (manual nkey gen + ACL grants in `nats-server.conf`). Tooling for this is a separate ticket if it surfaces.
- **Migration tooling** ‚Äî feature flag implementation may be a single env var (`LLMCLI_LIFECYCLE_VIA_NATS=1`) read by CLI; ¬¨a full migration framework.

## Premise Validity

**Success in 6 months:** `llmcli swap --host roxabituwer qwen3-4b` works from any tailnet member with a valid operator nkey. `src/llmcli/daemon.py` + the AF_UNIX socket no longer exist on disk. Every lifecycle action (swap, stop, reload-catalog) emits a JetStream-replayable record. M‚ÇÅ + M‚ÇÇ run a **single quadlet** per host (the worker), no host daemon, no supervisor wrapper for llmCLI lifecycle.

**Failure in 6 months (falsifiable):** Both control planes still coexist: `daemon.py` and lifecycle NATS subjects both present in `main`, gated by a feature flag (`LLMCLI_LIFECYCLE_VIA_NATS`) that was never flipped to default-on. Operators still SSH to the GPU host to swap models because the lifecycle path was never trusted enough to delete the fallback. Concretely: `git ls-files src/llmcli/daemon.py` returns the file 6 months from cutover-PR merge ‚Üí failure.

**Simplest alternative:** Option A from ADR-004 / closed issue #33 ‚Äî wrap `daemon.py` under supervisor on M‚ÇÇ (mirror M‚ÇÅ's `lyra.service` autostart linger). ~50 lines of `conf.d/llmcli_daemon.conf` + `make register` entry.

**Why not simplest:** Preserves both control planes. Lifecycle stays SSH-only because AF_UNIX is host-scoped. Two deploy patterns per host remain (quadlet + supervisor). Doesn't scale to a third GPU host ‚Äî every new host pays the duplication tax. ADR-004 already settled this: A relocates the boundary, doesn't remove the asymmetry.

## Complexity

**Tier: F-full** ‚Äî declared in issue body; confirmed by signals.

Signals observed:

- **Multi-domain:** contracts (cross-repo `Roxabi/lyra`) + worker (Python NATS adapter) + CLI (Typer commands) + deploy (Quadlet/systemd) + tests + docs. Domains: backend + infra.
- **New architecture:** queue-group-of-1 vs per-worker subjects (open question O1) introduces a NATS routing pattern llmCLI doesn't use today; `LifecycleMixin` is new code, not a refactor of `daemon.py`.
- **Fleet-wide impact:** every GPU host (M‚ÇÅ + M‚ÇÇ + future) and every operator surface.
- **Unknowns concentrated in 4 open questions** (auth split, hot-swap policy mid-stream, reload-catalog broadcast vs per-host, migration cutover strategy) ‚Äî spec must resolve.
- **Cross-repo coordination:** PRs to `Roxabi/lyra` staging for `roxabi-contracts.llm` (new subjects + models + error codes) precede or co-merge with llmCLI's PR.

## Open Questions (resolve in /analyze ‚Üí /spec)

| ID | Question | Default leaning (revisable) |
|---|---|---|
| O1 | **Swap atomicity:** queue-group-of-1 (would need SDK extension to `NatsAdapterBase`) vs per-worker-targeted subjects (`lyra.llm.lifecycle.swap.{worker_id}` via existing `_extra_subjects()` broadcast)? | **Per-worker subjects** ‚Äî no SDK change, operator routes via `--host`, fan-out `--fleet` uses wildcard. |
| O2 | **Auth split:** separate operator nkey (subject perm split: generate vs lifecycle) vs reuse worker nkey? | **Separate** ‚Äî reads cleanly off ADR-004 "Subject permissions cleanly split inference consumers from operators." |
| O3 | **Hot-swap mid-stream:** drain in-flight generations (wait N seconds, then hard-cut) vs hard-cut + return retryable error to streamers? | **Drain with timeout** ‚Äî matches `NatsAdapterBase.drain_timeout` precedent; tunable. |
| O4 | **Migration cutover:** feature-flag both paths in parallel (`LLMCLI_LIFECYCLE_VIA_NATS=1`) then flip + delete, vs hard switch in single PR? | **Feature-flag** ‚Äî safer rollout; cutover PR is one-liner default flip + daemon deletion. |
| O5 | **Reload-catalog scope:** broadcast `lyra.llm.lifecycle.reload-catalog` (every worker re-reads its TOML) vs per-host `.{worker_id}`? | **Broadcast** ‚Äî catalog changes are file-system-local; broadcast tells *every* host "your TOML may have changed, re-read." |
