## Problem

The Quadlet `llmcli proxy` (`:18091`) generates its YAML config from the catalog alone (`build_full_config` in `src/llmcli/litellm_config.py:79-83` emits only `master_key`, `drop_params`, `model_list`). - `litellm_settings.use_chat_completions_url_for_anthropic_messages: true` (Anthropic â†’ OpenAI auto-translate)

Secondary: the proxy port (`18091`) is hardcoded across `cli/proxy.py` default, the Quadlet `PublishPort=` line, and consumer URLs. The catalog `[host]` section is the natural home for it but doesn't carry it today.

## Constraints

- Backwards compat: hosts without a `proxy-base.yaml` file must still get a sensible default config (the current minimal shape from `build_full_config`).
- LiteLLM API stability: rely on documented YAML config, no monkey-patching. Upgrades to LiteLLM should require only a `vim ~/.roxabi/llmcli/proxy-base.yaml` edit, not llmCLI code changes.
- Container security: `proxy-base.yaml` is bind-mounted RO into the Quadlet (same pattern as the existing catalog mount).
- F-lite tier: ~5-6 files, single domain (backend Python).
- Env-var indirection for secrets: pass-through `Authorization: Bearer os.environ/FOO` (no inline secrets in the YAML).

## Out of Scope

- Mutating `proxy-base.yaml` from llmCLI tooling â€” file is purely user-owned (no sentinels, no managed block).
- Retiring `register-proxy` and the `~/.litellm/config.yaml` sentinel mechanism â€” they remain for backwards-compat with the legacy `:4000` supervisor program; their removal is a separate cleanup once `:4000` retires (#51).

`llmcli proxy` startup:

1. Load `~/.roxabi/llmcli/proxy-base.yaml` if present (else use built-in default with `master_key` + `drop_params`).
2. Load catalog.
3. Build `model_list` from catalog (existing logic, moved out of `build_full_config`).
4. Merge: deep-merge base YAML + computed `model_list` â†’ write to `~/.local/state/llmcli/proxy.config.yaml`.
5. Spawn `litellm --config <merged>` on `[host].port` (default 18091).

Quadlet bind-mount picks up `proxy-base.yaml` automatically (same path as existing catalog mount).

**Failure in 6 months (falsifiable):** Any one of:
1. `:4000` still active in lyra supervisor 30 days post-merge.
2. â‰¥1 alias in `~/.bash_aliases` still references `:4000`.
3. A LiteLLM transport feature (new pass-through, new `litellm_settings` key) required an llmCLI code change in those 30 days â€” proves the schema-translation worry was unfounded and we picked the wrong layer.

**Simplest alternative:** Bind-mount the existing `~/.litellm/config.yaml` (host path) into the Quadlet RO; have `llmcli proxy` skip its config generator and spawn `litellm --config <bind-mounted>`.

**Why not simplest:**
1. Couples the Quadlet runtime to a host file path (`~/.litellm/`) that conceptually belongs to the legacy lyra-supervisor world we are retiring.
2. Mâ‚پ has no `~/.litellm/`; we'd have to create one with hand-curated content â€” duplicating what `proxy-base.yaml` would do but in a less natural location.
3. The `register-proxy` sentinel-managed model_list block is harder to reason about than "two files, one merge"; the latter is the cleaner long-term shape.
